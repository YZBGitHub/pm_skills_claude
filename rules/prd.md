# PRD 撰写规范（PRD Writing Rules）

> 本文件定义 PRD 撰写的标准和质量要求，由 prd-specialist 和 design-specialist 遵循。
> **PRD 评审标准（六维 + Blocker/Major/Minor 分级）由 prd-reviewer 单独承载**，见 `skills/prd-reviewer/SKILL.md`。本文件不再包含"自评审清单"。

## 写作原则

- **精确胜于模糊**: "输入框最多50个字符" 优于 "有长度限制"
- **规则胜于描述**: 用明确的业务规则代替笼统的功能描述
- **表格胜于段落**: 结构化信息优先使用表格呈现
- **可测试**: 每个验收标准必须可以在原型中验证

## PRD.md 文档头模板

```markdown
| 文档信息 | |
|---------|---|
| 项目名称 | xxx |
| 文档版本 | v1.0 |
| 创建日期 | yyyy-mm-dd |
| 文档状态 | 草稿 / 待评审 / 评审中 / 修订中 / 评审通过 / 已定稿 |
```

## 章节填充要求

### 第 1-2 章（背景+用户）
- 项目目标必须可量化（如"学生选课完成率 ≥95%"）
- 用户画像至少 3 类：学生 / 教师 / 教务
- 场景使用"触发条件→前置条件→主流程→异常流程→预期结果"模板

### 第 3 章（功能需求）
- 每个功能必须有：功能描述、优先级（P0/P1/P2）、用户角色、字段规格表、业务规则
- 字段规格表必须包含：字段名 / 类型 / 必填 / 校验规则 / 说明
- 状态流转用"初始态 → 触发条件 → 目标态"表达

### 第 4 章（非功能需求）
- 性能指标（如首屏 ≤2s）
- 安全需求（鉴权、数据脱敏）
- 兼容性（Chrome 80+、微信内置浏览器、PC 1920×1080 / 移动 375×812）

### 第 5 章（数据埋点）
- 埋点字段：页面/操作 / 事件名 / 参数 / 用途

### 第 6 章（边界 Case）
- 至少覆盖：空状态、满状态、超时、权限不足、网络异常 5 种情况
- 每种情况给出明确的"用户提示"文案

### 第 7 章（用户故事）
- 格式：`作为 <具体角色>，我想要 <行为>，以便 <价值>`
- 遵循 INVEST 原则（Independent / Valuable / Small / Testable）
- 验收标准用 Given-When-Then，**必须具体**（不能写"Then 系统正确处理"）
- 编号：`US-<模块缩写>-<三位数字>`

### 第 8 章（设计风格 + 交互 + 视觉）

第 8 章分为 6 个子节，顺序不可乱：

| 子节 | 标题 | 内容 |
|------|------|------|
| 8.1 | 设计风格定调 | 选定风格流派（A-E 或品牌 VIS）、关键词、主色方向、字体/圆角/图标/动效基调、反风格约束 |
| 8.2 | 信息架构 | 树状页面结构 + 导航结构（PC/移动） |
| 8.3 | 交互规范 | 页面类型与布局 + 通用交互模式（加载/空状态/反馈/危险/长列表/搜索） |
| 8.4 | 视觉规范 | 配色、字体、间距、圆角、阴影（数值必须与 8.1 风格一致） |
| 8.5 | 响应式规范 | Desktop ≥1200 / Tablet 768-1199 / Mobile <768，触摸目标 ≥44×44px |
| 8.6 | 组件规范 | 按钮 / 输入框 / 表格 / 弹窗等的 PC/移动端尺寸 |

**关键约束**：8.4-8.6 的所有数值必须服从 8.1 已定调的设计风格。若冲突，先回头调整 8.1 再往下写。

### 预设设计风格（8.1 候选集）

| 代号 | 风格 | 典型场景 |
|------|------|---------|
| A | 简约现代 | 教务 / 成绩 / 后台管理（高频事务） |
| B | 教育稳重 | 招生门户 / 校务公告 / 对外官方系统 |
| C | 青春活力 | 学生端 App / 校园社区 / 迎新 / 社团 |
| D | 科技未来 | 数据大屏 / 智慧校园 / 实训 3D 展示 |
| E | 友好温暖 | 心理健康 / 就业服务 / 学业辅导 |

具体参数（配色、字体、圆角、图标、动效）详见 `skills/design-specialist/SKILL.md` 阶段二。

## 撰写自检（撰写时的最低基线）

> 这不是评审清单。评审在独立的 prd-reviewer 角色完成。本节仅作为 prd-specialist 在落笔时的"提交前最低基线"，避免明显粗糙的草稿浪费 reviewer 的轮次。

提交评审前，prd-specialist 应自检：

- [ ] 大纲所有条目都已映射到 PRD 章节，没有"漏写"
- [ ] 每个功能写明：用户角色、字段规格、业务规则
- [ ] 文档信息表头已填（项目名 / 版本 / 日期 / 状态 = "待评审"）
- [ ] 所有章节标题与 rules/workflow.md 的章节图编号一致
- [ ] 没有占位符 / TBD（如有未确定项，明确标记 `待补充（原因：...）`）

**注意**：自检 ≠ 自评审。自检只确认"是否到了能提交评审的状态"，不替代 prd-reviewer 的独立审查。
